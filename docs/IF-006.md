## CONTEXTO

Analyxa hasta ahora analiza archivos individuales. Con esta IF se convierte en un sistema que:

1. **Recibe conversaciones por cola** (Redis) — un agente de IA pushea conversaciones, Analyxa las procesa
2. **Almacena análisis como vectores** (Qdrant) — búsqueda semántica por similitud entre sesiones
3. **Procesa lotes** (Batch) — analizar N conversaciones de golpe desde archivos o Redis
4. **Docker Compose** — Redis + Qdrant levantados de un comando

Pipeline producción: `Redis → Batch → Analyzer → Qdrant`

**Conexión:** RedisSource alimenta al Analyzer. QdrantSink recibe el AnalysisResult. BatchProcessor orquesta múltiples análisis. CLI extiende con 3 comandos nuevos.

### Archivo 2: `src/analyxa/sources/redis_source.py`

Lector de conversaciones desde Redis. Patrón: lista como cola + hash para metadata + marcado de procesadas.

**Esquema de Redis:**
- `analyxa:conversations` — Lista (cola FIFO). Cada elemento es un conversation_id.
- `analyxa:conv:{id}` — Hash con keys: `text` (conversación), `schema` (schema a usar), `context` (JSON string opcional), `status` ("pending" | "analyzed" | "failed"), `pushed_at` (ISO timestamp).

**Clase: `RedisSource`**

```
__init__(self, url: str | None = None)
  - url: URL de Redis. Si None, leer de config (config.redis_url) o default "redis://localhost:6379"
  - Inicializar redis.Redis.from_url(url, decode_responses=True)
  - Si conexión falla → NO lanzar error en init. Lazy connection.
```

```
push(self, conversation: str, conversation_id: str | None = None, schema: str = "universal", context: dict | None = None) -> str
  - Si conversation_id es None → generar UUID4
  - Crear hash analyxa:conv:{id} con:
    - text: conversation
    - schema: schema
    - context: json.dumps(context) si context else ""
    - status: "pending"
    - pushed_at: datetime.now(timezone.utc).isoformat()
  - Push conversation_id a la lista analyxa:conversations (rpush → cola FIFO)
  - Retornar conversation_id
```

```
get(self, conversation_id: str) -> dict | None
  - Leer hash analyxa:conv:{id}
  - Si no existe → retornar None
  - Parsear context de JSON string a dict (si no vacío)
  - Retornar dict con: id, text, schema, context, status, pushed_at
```

```
pending(self) -> list[str]
  - Obtener todos los IDs de la lista analyxa:conversations
  - Filtrar solo los que tienen status="pending" en su hash
  - Retornar lista de conversation_ids pendientes
```

```
next(self) -> dict | None
  - lpop de analyxa:conversations (saca el primero de la cola)
  - Si None → no hay conversaciones en cola, retornar None
  - Leer el hash de ese conversation_id
  - Si status no es "pending" → descartar y llamar next() recursivamente (max 100 intentos para evitar loop infinito)
  - Retornar dict con: id, text, schema, context, status, pushed_at
```

```
mark_analyzed(self, conversation_id: str) -> None
  - Setear status="analyzed" en hash analyxa:conv:{id}
  - Setear analyzed_at=datetime.now(timezone.utc).isoformat()
```

```
mark_failed(self, conversation_id: str, error: str) -> None
  - Setear status="failed" en hash analyxa:conv:{id}
  - Setear error=error
  - Setear failed_at=datetime.now(timezone.utc).isoformat()
```

```
list_all(self, status: str | None = None) -> list[dict]
  - Buscar todas las keys que matcheen analyxa:conv:* (usar scan_iter, NO keys *)
  - Si status filtro → solo retornar las que matcheen
  - Retornar lista de dicts con id, schema, status, pushed_at
  - Ordenar por pushed_at descendente
```

```
flush(self) -> int
  - Eliminar todas las keys analyxa:* (conversaciones + lista)
  - Retornar número de keys eliminadas
  - Para testing/reset
```

**Imports:** `redis`, `json`, `uuid`, `datetime`
Import condicional de redis (mismo patrón que anthropic/openai en llm_client).

### Archivo 3: `src/analyxa/sinks/qdrant_sink.py`

Almacenador de análisis en Qdrant como vectores.

**Clase: `QdrantSink`**

```
__init__(self, url: str | None = None, collection: str | None = None)
  - url: URL de Qdrant. Si None → config.qdrant_url o "http://localhost:6333"
  - collection: nombre de la colección. Si None → config.qdrant_collection o "analyxa_analyses"
  - Inicializar qdrant_client.QdrantClient(url=url)
  - Llamar _ensure_collection() para crear la colección si no existe
```

```
_ensure_collection(self) -> None
  - Verificar si la colección existe (client.collection_exists)
  - Si no existe → crearla con:
    - vectors_config: size=1536, distance=Cosine
  - Si ya existe → no hacer nada
```

```
store(self, result: "AnalysisResult") -> str
  - Si result.embedding es None → generar un ID pero almacenar sin vector (usar punto con vector dummy o skip — ver nota abajo)
  - Generar point_id como UUID string
  - Crear payload desde result.to_dict() (sin el embedding, que va como vector)
  - Upsert point:
    - id: UUID
    - vector: result.embedding (1536 floats)
    - payload: dict con todos los campos + _meta
  - Retornar point_id
  
  NOTA: Si no hay embedding, almacenar con vector de zeros [0.0]*1536 y agregar _meta.has_embedding=False en payload. Qdrant requiere vector para cada punto. El flag permite filtrar en búsquedas.
```

```
search_similar(self, query_embedding: list[float], limit: int = 10, filters: dict | None = None) -> list[dict]
  - Buscar los `limit` puntos más similares por cosine distance
  - Si filters → construir qdrant Filter:
    Ejemplo: {"sentiment": "negative"} → FieldCondition(key="sentiment", match=MatchValue(value="negative"))
    Ejemplo: {"schema_name": "support"} → filter por _meta.schema_name
  - Retornar lista de dicts con: id, score, payload (los campos del análisis)
```

```
get(self, point_id: str) -> dict | None
  - Retrieve punto por ID
  - Retornar payload completo si existe, None si no
```

```
count(self, filters: dict | None = None) -> int
  - Contar puntos en la colección (con filtros opcionales)
```

```
delete_collection(self) -> None
  - Eliminar la colección completa
  - Para testing/reset
```

**Imports:** `qdrant_client`, `qdrant_client.models` (Distance, VectorParams, PointStruct, Filter, FieldCondition, MatchValue)
Import condicional de qdrant_client.
`uuid` para generar IDs.

### Archivo 4: `src/analyxa/batch.py`

Procesador de lotes. Analiza múltiples conversaciones en secuencia.

**Dataclass: `BatchResult`**

```python
@dataclass
class BatchResult:
    total: int                         # Total de conversaciones procesadas
    successful: int                    # Análisis exitosos
    failed: int                        # Análisis fallidos
    results: list["AnalysisResult"]    # Lista de resultados individuales
    errors: list[dict]                 # Lista de {id, error} para los fallidos
    elapsed_seconds: float             # Tiempo total del batch
    
    @property
    def success_rate(self) -> float:
        return self.successful / self.total if self.total > 0 else 0.0
```

**Función: `batch_analyze`**

```
batch_analyze(
    conversations: list[dict],
    schema_name: str = "universal",
    provider: str = "anthropic",
    model: str | None = None,
    enable_embeddings: bool = True,
    sink: Any | None = None,
    on_progress: Callable | None = None
) -> BatchResult
```

**Parámetros:**
- `conversations`: lista de dicts con al menos `{"text": str}`. Opcionalmente `{"text": str, "id": str, "context": dict}`
- `sink`: si se provee, cada resultado se escribe al sink (QdrantSink, JsonSink, etc.) inmediatamente después de analizar
- `on_progress`: callback `(current: int, total: int, result: AnalysisResult | None) -> None` para reportar progreso

**Lógica:**
1. Crear UN solo Analyzer (reutilizar para todos los análisis — reusa cache de schema)
2. Iterar sobre conversations:
   - Extraer text, id (default: index), context
   - Llamar analyzer.analyze(text, context)
   - Si sink → sink.store(result) o sink.write(result.to_flat_dict())
   - Si on_progress → llamar callback
   - Registrar éxito o fallo
3. Construir y retornar BatchResult

**Función: `batch_analyze_from_redis`**

```
batch_analyze_from_redis(
    redis_source: "RedisSource | None" = None,
    qdrant_sink: "QdrantSink | None" = None,
    max_items: int | None = None,
    provider: str = "anthropic",
    model: str | None = None,
    enable_embeddings: bool = True,
    on_progress: Callable | None = None
) -> BatchResult
```

**Lógica:**
1. Si redis_source es None → crear con defaults
2. Si qdrant_sink es None → crear con defaults
3. Obtener conversaciones pendientes de Redis (pending() o next() en loop)
4. Si max_items → limitar cantidad
5. Para cada conversación:
   - Crear Analyzer con el schema indicado en la conversación de Redis
   - Analizar
   - Si éxito → mark_analyzed + store en Qdrant
   - Si fallo → mark_failed
   - Si on_progress → callback
6. Retornar BatchResult

**Imports:** `time`, `dataclasses`, `typing`

## NOTAS PARA CLAUDE CODE

1. **DISCIPLINA:** Esta IF tiene Docker, Redis, Qdrant, batch, CLI y tests. Sigue el orden exacto: (1) docker-compose, (2) levantar servicios, (3) redis_source, (4) qdrant_sink, (5) batch, (6) CLI, (7) pyproject, (8) tests. Si un paso falla, no avanzar al siguiente.

2. **Docker compose PRIMERO.** Sin Redis y Qdrant corriendo, los tests de integración fallan. Verificar con `docker compose ps` antes de escribir código.

3. **Redis keys tienen prefijo `analyxa:`.** No contaminar el namespace. Usar `scan_iter(match="analyxa:conv:*")` para listar, NUNCA `keys *` (bloquea Redis en producción).

4. **Qdrant auto-create collection.** `_ensure_collection()` se llama en el constructor. Si la colección ya existe, no hacer nada. Usar `client.collection_exists()` antes de crear.

5. **Vectores de zeros para puntos sin embedding.** Qdrant requiere un vector para cada punto. Si el análisis no tiene embedding (OpenAI no configurada), usar `[0.0] * 1536` y marcar `has_embedding=False` en payload. Los searches filtran por este flag.

6. **batch_analyze crea UN solo Analyzer** y lo reutiliza. No crear un Analyzer nuevo para cada conversación — eso recarga el schema cada vez.

7. **Tests de infra son INTEGRATION tests.** Requieren Docker corriendo. Si pytest detecta que Redis/Qdrant no están disponibles, los tests deben SKIPPEARSE con `pytest.mark.skipif`, no fallar. Usar fixture que verifica conexión.

8. **CLI imports son LAZY.** Los comandos redis/batch/search importan RedisSource/QdrantSink/batch dentro de la función, no al top del archivo. `analyxa --help` no debe fallar si Redis no está instalado.

9. **flush() es DESTRUCTIVO.** El comando CLI `analyxa redis flush` tiene `@click.confirmation_option`. Los tests usan flush en teardown fixture.

10. **Las API keys (Anthropic/OpenAI) siguen pendientes en .env.** Esta IF no las necesita para tests (todo mockeado o infra-only). Se necesitarán en IF-007 para el dogfooding real.
