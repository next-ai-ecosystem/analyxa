## CONTEXTO

Esta es la PRIMERA instrucción de fabricación de Analyxa. El Schema System es el cimiento de todo el motor: define cómo se configuran las dimensiones que se extraen de cada conversación. Sin schemas, no hay prompts, no hay análisis, no hay producto.

Se construyen 3 cosas:
1. **schema.py** — El SchemaManager que carga, valida, hereda y cachea schemas YAML
2. **universal.yaml** — El schema base con 10 campos que todo análisis incluye
3. **support.yaml** — El primer schema vertical (hereda universal + 6 campos propios)

Más tests para verificar que todo funciona.

### Archivo 2: `src/analyxa/schemas/universal.yaml`

Schema base universal. 4 secciones obligatorias:

**Sección 1 — metadata:**
```yaml
metadata:
  name: universal
  version: "1.0"
  description: "Universal schema — 10 base fields for any conversation type"
  inherits: null   # No hereda de nadie, es la raíz
```

**Sección 2 — fields:** (lista ordenada de 10 campos)

Cada campo tiene esta estructura:
```yaml
fields:
  - name: [nombre]
    type: [string | keyword | keyword_array | string_array | number | boolean]
    required: [true | false]
    description: [descripción corta para documentación]
    prompt_hint: [instrucción detallada para el LLM — 1-3 oraciones que explican exactamente qué extraer y cómo]
    allowed_values: [lista de valores permitidos, solo para keyword y keyword_array, null para tipos abiertos]
```

Los 10 campos con sus specs:

| # | name | type | required | allowed_values | prompt_hint (guía) |
|---|------|------|----------|----------------|-------------------|
| 1 | title | string | true | null | Nombre descriptivo de la sesión en 5-8 palabras. Captura el tema principal y resultado. |
| 2 | summary | string | true | null | Resumen de 3-5 oraciones cubriendo: qué pidió el usuario, qué hizo el agente, resultado final. Este campo se vectoriza para búsqueda semántica, incluir detalles relevantes. |
| 3 | sentiment | keyword | true | [positive, negative, mixed, neutral] | Sentimiento predominante del USUARIO (no del agente) a lo largo de toda la conversación. |
| 4 | sentiment_intensity | keyword | true | [low, medium, high] | Qué tan fuerte se manifiesta el sentimiento. Low = sutil/implícito, medium = claro, high = explícito/emocional. |
| 5 | topics | keyword_array | true | null | Lista de 1-5 temas específicos discutidos. Usar términos concretos (ej: "billing", "refund"), NO genéricos (ej: "problema", "consulta"). |
| 6 | session_outcome | keyword | true | [resolved, unresolved, escalated, abandoned] | Resultado final de la sesión. Resolved = el usuario obtuvo lo que necesitaba. Abandoned = el usuario dejó la conversación sin conclusión. |
| 7 | user_intent | string | true | null | En 1-2 oraciones, qué buscaba realmente el usuario. Distinguir entre lo que pidió explícitamente y lo que realmente necesitaba. |
| 8 | risk_signals | keyword_array | false | [frustration, churn_risk, complaint, urgency, confusion, escalation_request, negative_experience, repeat_contact] | Señales de riesgo detectadas. Lista vacía si no hay señales. Solo incluir señales con evidencia clara en el texto. |
| 9 | key_entities | keyword_array | false | null | Personas, productos, servicios, fechas, montos mencionados explícitamente. No inferir — solo extraer lo que aparece en el texto. |
| 10 | action_items | string_array | false | null | Compromisos explícitos o siguientes pasos mencionados. Cada ítem debe ser accionable y específico. Lista vacía si no hay. |

**Sección 3 — auto_fields:**

Campos que NO requieren LLM, se calculan automáticamente:
```yaml
auto_fields:
  - name: schema_name
    source: config
    description: "Name of the schema used for analysis"
  - name: schema_version
    source: config
    description: "Version of the schema used"
  - name: analyzed_at
    source: timestamp
    description: "ISO 8601 timestamp of analysis"
  - name: analysis_model
    source: config
    description: "LLM model used for extraction"
  - name: embedding_model
    source: config
    description: "Embedding model used for vectorization"
  - name: session_length
    source: computed
    description: "Number of messages in the conversation"
  - name: conversation_hash
    source: computed
    description: "SHA-256 hash (first 16 chars) for deduplication"
```

**Sección 4 — prompt:**

Instrucciones globales para el LLM cuando usa este schema:
```yaml
prompt:
  system_context: >
    You are a conversation analysis engine. Your task is to extract structured
    information from conversations between users and AI agents. Be precise,
    evidence-based, and avoid speculation. Extract only what is clearly present
    or strongly implied in the conversation text.
  output_format: >
    Respond with a single valid JSON object. No markdown, no code fences,
    no explanatory text. Every required field must be present. Optional fields
    should be included only if there is clear evidence in the conversation.
  analysis_guidelines:
    - Focus on the USER's experience, not the agent's responses
    - Sentiment reflects the user's emotional state, not the conversation topic
    - Topics must be specific and actionable, not generic labels
    - Risk signals require evidence — do not infer from absence of positivity
    - Entity extraction is literal — only extract what appears in the text
```

### Archivo 4: `src/analyxa/schema.py`

El SchemaManager. Componente central que gestiona schemas.

**Clase: SchemaManager**

Constructor:
```
__init__(self, schemas_dir: str | Path | None = None)
  - schemas_dir: directorio donde buscar archivos .yaml
  - Default: buscar en el directorio schemas/ relativo al paquete (src/analyxa/schemas/)
  - Atributo _cache: dict vacío para cache de schemas ya cargados
  - Atributo _schemas_dir: Path resuelto
```

Método: load_schema
```
load_schema(self, name: str) -> dict
  - Recibe nombre del schema (sin .yaml), ej: "universal", "support"
  - Si está en cache → retornar del cache
  - Buscar archivo {name}.yaml en schemas_dir
  - Si no existe → raise FileNotFoundError con mensaje claro
  - Cargar con yaml.safe_load()
  - Validar estructura (debe tener 'metadata' y 'fields')
  - Si tiene inherits en metadata:
      1. Cargar el schema padre (recursivo)
      2. Resolver herencia (campos del padre + campos propios)
      3. Merge de auto_fields y prompt del padre (si el hijo no los tiene)
  - Guardar en cache
  - Retornar schema completo (ya con herencia resuelta)
```

Método: resolve_inheritance (privado)
```
_resolve_inheritance(self, child: dict, parent: dict) -> dict
  - Crear nuevo dict
  - metadata: la del hijo (no merge)
  - fields: campos del PADRE primero + campos del HIJO después (orden importa para el prompt)
  - auto_fields: los del hijo si existen, si no los del padre
  - prompt: los del hijo si existen, si no los del padre
  - Retornar schema resuelto
```

Método: list_schemas
```
list_schemas(self) -> list[str]
  - Listar todos los .yaml en schemas_dir
  - Retornar lista de nombres (sin extensión), ordenada alfabéticamente
```

Método: get_field_names
```
get_field_names(self, name: str) -> list[str]
  - Cargar schema (usa cache)
  - Retornar lista de nombres de campos en orden
```

Método: validate_result
```
validate_result(self, name: str, result: dict) -> tuple[bool, list[str]]
  - Cargar schema
  - Verificar que todos los campos required están presentes en result
  - Verificar que los campos keyword tienen valores dentro de allowed_values (si están definidos)
  - Verificar tipos básicos: keyword→str, keyword_array→list, boolean→bool, etc.
  - Retornar (is_valid, list_of_errors)
  - Si is_valid=True, errors es lista vacía
```

**Manejo de errores:**
- FileNotFoundError si schema no existe
- ValueError si YAML es inválido o falta metadata/fields
- Mensajes de error descriptivos siempre

## NOTAS PARA CLAUDE CODE

1. **No inventar campos adicionales.** Los 10 campos universales y 6 de soporte están definidos arriba. No agregar más.
2. **El orden de campos importa.** En herencia, los campos del padre van PRIMERO, luego los del hijo. El prompt se genera en este orden.
3. **YAML safe_load siempre.** Nunca yaml.load() — riesgo de seguridad.
4. **Los prompt_hints son la salsa secreta.** Son las instrucciones que el LLM va a recibir para cada campo. Deben ser claros, específicos y accionables. Copiarlos fielmente del spec de arriba.
5. **El directorio default de schemas** debe resolverse relativo al paquete (Path(__file__).parent / "schemas"), no relativo al cwd.
6. **Si el workspace no existe**, crearlo con toda la estructura antes de empezar con el código.
7. **Activar venv** antes de cualquier pip install o pytest.
