## CONTEXTO

El Schema System (IF-001) está operativo: SchemaManager carga, hereda y valida schemas YAML. Ahora necesitamos el componente que **convierte un schema + una conversación en un prompt listo para enviar al LLM**.

El Prompt Generator es el traductor entre la configuración declarativa (YAML) y la instrucción ejecutable (prompt). Recibe un schema cargado y el texto de una conversación, y produce un par `{system_prompt, user_prompt}` que el LLM Client (IF-003) consumirá directamente.

**Principio clave:** Los prompts se generan DINÁMICAMENTE desde el schema. Si mañana se agrega un campo nuevo al YAML, el prompt lo incluye automáticamente sin tocar código.

**Conexión con lo anterior:** Usa `SchemaManager.load_schema()` para obtener el schema completo con herencia resuelta.

**Conexión con lo siguiente:** El LLM Client (IF-003) recibirá el output de `build_prompt()` y lo enviará a Claude Sonnet / GPT-4o.

### Archivo 1: `src/analyxa/prompt_builder.py`

Módulo con una función principal y helpers internos.

---

**Función principal: `build_prompt`**

```
build_prompt(schema: dict, conversation: str, context: dict | None = None) -> dict
```

**Parámetros:**
- `schema`: dict — schema completo ya cargado (con herencia resuelta). Es el output de `SchemaManager.load_schema()`.
- `conversation`: str — texto crudo de la conversación a analizar.
- `context`: dict | None — contexto adicional opcional. Puede incluir:
  - `agent_name`: str — nombre del agente (ej: "SupportBot")
  - `agent_type`: str — tipo de agente (ej: "customer_support")
  - `company`: str — nombre de la empresa
  - `language`: str — idioma esperado de la respuesta (default: "en")
  - Cualquier otro key:value que el usuario quiera inyectar

**Retorna:** dict con exactamente dos keys:
```python
{
    "system": str,   # System prompt completo
    "user": str      # User prompt completo
}
```

---

**Lógica del SYSTEM prompt:**

El system prompt se construye concatenando estas secciones en orden:

**Sección 1 — Rol base:**
Usar el `prompt.system_context` del schema. Si no existe, usar default:
```
"You are a conversation analysis engine. Your task is to extract structured information from conversations between users and AI agents. Be precise, evidence-based, and avoid speculation."
```

**Sección 2 — Contexto adicional (solo si `context` no es None):**
```
"Additional context for this analysis:
- Agent name: {agent_name}
- Agent type: {agent_type}
..."
```
Solo incluir las keys que existan en context. Omitir keys con valor None.

**Sección 3 — Definición de campos:**
Generar dinámicamente desde `schema['fields']`:
```
"You must extract the following fields:

1. title (string, required): Nombre descriptivo de la sesión en 5-8 palabras...
   [aquí va el prompt_hint del campo]

2. summary (string, required): Resumen de 3-5 oraciones...
   [prompt_hint]

...

10. action_items (string_array, optional): Compromisos explícitos...
   [prompt_hint]
```

Para cada campo incluir:
- Número de orden (1, 2, 3...)
- name
- type entre paréntesis
- "required" o "optional" entre paréntesis
- description
- En nueva línea indentada: el prompt_hint completo
- Si tiene allowed_values: `Allowed values: [val1, val2, val3]`

**Sección 4 — Formato de output:**
Usar el `prompt.output_format` del schema. Si no existe, usar default:
```
"Respond with a single valid JSON object. No markdown, no code fences, no explanatory text before or after the JSON. Every required field must be present. Optional fields should be included only if there is clear evidence."
```

**Sección 5 — Guidelines adicionales:**
Si el schema tiene `prompt.analysis_guidelines` (lista de strings), incluirlas:
```
"Analysis guidelines:
- Focus on the USER's experience, not the agent's responses
- Sentiment reflects the user's emotional state...
- ..."
```

**Sección 6 — Template JSON:**
Generar un template JSON con los nombres de los campos y placeholders de tipo:
```
"Your response must follow this exact JSON structure:
{
  "title": "string",
  "summary": "string",
  "sentiment": "positive|negative|mixed|neutral",
  "sentiment_intensity": "low|medium|high",
  "topics": ["keyword1", "keyword2"],
  "session_outcome": "resolved|unresolved|escalated|abandoned",
  "user_intent": "string",
  "risk_signals": ["signal1", "signal2"],
  "key_entities": ["entity1", "entity2"],
  "action_items": ["item1", "item2"]
}"
```

Para generar el placeholder de cada campo:
- `string` → `"string"`
- `keyword` con allowed_values → `"val1|val2|val3"`
- `keyword` sin allowed_values → `"keyword"`
- `keyword_array` con allowed_values → `["val1", "val2"]`
- `keyword_array` sin allowed_values → `["keyword1", "keyword2"]`
- `string_array` → `["string1", "string2"]`
- `number` → `0`
- `boolean` → `true`

---

**Lógica del USER prompt:**

Más simple. Se construye así:

```
"Analyze the following conversation and extract all fields as specified.

CONVERSATION:
---
{conversation}
---

Respond with the JSON object only."
```

Si `context` tiene key `language` y no es "en":
```
"Analyze the following conversation and extract all fields as specified.
Respond in {language} for all free-text fields (title, summary, user_intent, action_items). Use English for keyword fields.

CONVERSATION:
---
{conversation}
---

Respond with the JSON object only."
```

---

**Helper interno: `_format_field_definition`**

```
_format_field_definition(index: int, field: dict) -> str
```

Recibe el número de orden y el dict de un campo. Retorna string formateado:
```
"3. sentiment (keyword, required): Sentimiento predominante del USUARIO...
   Hint: Sentimiento predominante del USUARIO (no del agente) a lo largo de toda la conversación.
   Allowed values: positive, negative, mixed, neutral"
```

---

**Helper interno: `_generate_json_template`**

```
_generate_json_template(fields: list[dict]) -> str
```

Recibe la lista de fields del schema. Retorna un JSON string formateado (con indentación) usando los placeholders por tipo descritos arriba.

IMPORTANTE: Este template debe ser JSON válido si se reemplazaran los placeholders. Usar `json.dumps()` con `indent=2` para generarlo, NO string concatenation manual.

Proceso:
1. Crear dict con key=field['name'], value=placeholder según tipo
2. `json.dumps(template_dict, indent=2)`
3. Retornar el string

---

**Helper interno: `_format_context`**

```
_format_context(context: dict) -> str
```

Recibe el dict de context. Retorna string formateado:
```
"Additional context for this analysis:
- Agent name: SupportBot
- Company: Acme Corp"
```

Solo incluir keys con valores no-None. Formatear keys con espacios y capitalización (snake_case → Title Case).

## NOTAS PARA CLAUDE CODE

1. **El prompt se genera DINÁMICAMENTE.** No hardcodear nombres de campos. Iterar sobre `schema['fields']` para generar cada sección.
2. **JSON template con json.dumps().** No construir JSON a mano con strings — es frágil y produce errores de formato. Crear un dict Python y serializar con `json.dumps(indent=2)`.
3. **Los prompt_hints son copiados TAL CUAL del schema.** No parafrasear ni resumir. Son instrucciones precisas para el LLM.
4. **El system prompt puede ser largo (2000+ chars para support con 16 campos).** Eso es correcto y esperado. Los LLMs manejan prompts largos bien.
5. **Separar secciones del system prompt con líneas en blanco** para legibilidad. El LLM los interpreta mejor con separación visual clara.
6. **No importar nada de IF-003 (llm_client).** prompt_builder es independiente — solo depende de schema.py.
7. **El context dict es pass-through.** No validar las keys — cualquier key:value que el usuario pase se incluye. Eso permite extensibilidad futura sin cambiar código.
